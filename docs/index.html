📖 Einführung

localSupabaseDB ist ein Template-Repository, das jeder WAMOCON Next.js-App als eigenständige, vollständig isolierte lokale Supabase-Instanz dient. Statt einer geteilten Cloud-Datenbank bekommt jede App ihren eigenen PostgreSQL-Container, Studio-Port und anon/service-Key – vollautomatisch verwaltet durch zwei PowerShell-Skripte.

1 Schnellstart

Für alle, die sofort loslegen möchten – ohne die Details zu lesen:

  1. Repository als Supabase-Verzeichnis einer App klonen
    # Klone das Repo als supabase/-Ordner in deine Next.js-App
git clone https://github.com/Wamocon/localSupabaseDB.git supabase
cd deine-nextjs-app
  2. Voraussetzungen sicherstellen – Docker Desktop läuft, Supabase CLI installiert
    supabase --version   # ≥ 2.78
docker ps            # Docker muss laufen
  3. Erstmalig starten 
    # Windows (PowerShell)
.\scripts\setup.ps1 -App meineapp

# macOS / Linux
./scripts/setup.sh --app meineapp
  4. Fertig! Das Skript gibt alle URLs und Keys aus und schreibt .env.local automatisch.
Ausgabe nach erfolgreichem Start

Du siehst drei Werte: NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY und SUPABASE_SERVICE_ROLE_KEY – bereits in .env.local deiner Next.js-App eingetragen. Studio ist erreichbar unter http://127.0.0.1:<port>.

3 Konzept

Jede Next.js-App bei WAMOCON kopiert dieses Repository als supabase/-Verzeichnis. Damit sind alle Supabase-Konfigurationen, Migrations und Seeds pro App eigenständig versioniert.

App-Isolation

Die Isolation wird über die project_id in der config.toml gesteuert. Das Setup-Skript setzt diese automatisch auf den gewählten App-Namen:

ResourceNamensschemaBeispiel (App: auktivo)
Docker-Volumesupabase_db_<app>supabase_db_auktivo
Containersupabase_<service>_<app>supabase_db_auktivo
config.tomlproject_id = "<app>"project_id = "auktivo"
env.localIm Root der Next.js-AppNEXT_PUBLIC_SUPABASE_URL=…

Port-Verwaltung

Ports werden automatisch pro App in der Datei .ports (JSON) gespeichert. So können mehrere Apps gleichzeitig laufen – jede auf ihrem eigenen Port-Block:

ServiceBeschreibungStandard (erste App)
apiPostgREST / Kong Gateway54321
dbPostgreSQL direkt54322
studioSupabase Studio UI54323
inbucketLokaler E-Mail-Posteingang54324
analyticsLogflare Analytics54325

Docker-Volumes & Datenpersistenz

PostgreSQL-Daten werden in einem benannten Docker-Volume gespeichert und überleben das Stoppen der Container. Ein -Reset löscht das Volume – alle Daten, Migrations und Seeds werden beim nächsten Start neu eingespielt.

Volume-Benennung

Das Volume heißt immer supabase_db_<appname>. Beim Wechsel der project_id ohne -Reset erkennt das Skript automatisch einen PostgreSQL-Versions-Konflikt und bricht mit einer hilfreichen Fehlermeldung ab.

4 Erstes Setup

Repository in eine App einbinden

Dieses Repository wird nicht als eigenständiges Projekt genutzt. Es wird als supabase/-Unterordner in jede Next.js-App geklont:

# Im Wurzelverzeichnis deiner Next.js-App ausführen
git clone https://github.com/Wamocon/localSupabaseDB.git supabase

# Abhängigkeiten installieren (Supabase CLI lokal)
cd supabase
npm install

Erstmalig starten

# Windows – PowerShell im supabase/-Verzeichnis
.\scripts\setup.ps1 -App meineapp

# macOS / Linux – im supabase/-Verzeichnis
./scripts/setup.sh --app meineapp

Das Skript führt automatisch folgende Schritte aus:

  • Prüft, ob Docker und Supabase CLI verfügbar sind
  • Setzt die project_id in der config.toml
  • Weist der App einen freien Port-Block zu (gespeichert in .ports)
  • Startet alle Supabase-Dienste (PostgreSQL 17, Studio, Auth, Storage, …)
  • Spielt Migrations aus supabase/migrations/ ein
  • Führt seed.sql aus
  • Liest anon key und service_role key aus
  • Schreibt .env.local in das Wurzelverzeichnis der Next.js-App

.env.local – generierte Variablen

VariableVerwendung
NEXT_PUBLIC_SUPABASE_URLAPI-Endpunkt für Client-seitigen Zugriff
NEXT_PUBLIC_SUPABASE_ANON_KEYÖffentlicher JWT-Schlüssel (mit RLS)
SUPABASE_SERVICE_ROLE_KEYAdmin-Schlüssel (nur serverseitig verwenden!)
Sicherheitshinweis

Den SUPABASE_SERVICE_ROLE_KEY niemals in Client-Code (NEXT_PUBLIC_) exponieren. Er umgeht Row Level Security und gewährt vollen Datenbankzugriff.

5 App starten & stoppen

Starten

# App mit bekanntem Namen starten (Ports werden wiederverwendet)
.\scripts\setup.ps1 -App auktivo

# App aus bestehender config.toml starten (app-Name aus project_id)
.\scripts\setup.ps1

Stoppen

# Supabase-Dienste stoppen (Daten bleiben im Volume erhalten)
.\scripts\stop.ps1 -App auktivo
Daten bleiben erhalten

Das Stoppen löscht keine Daten. PostgreSQL-Daten liegen im Docker-Volume und stehen beim nächsten Start sofort wieder zur Verfügung.

Automatische Retry-Logik

Das Setup-Skript versucht beim Start bis zu 3 Mal, die Dienste hochzufahren:

VersuchAktion
1Normaler supabase start
2Stop + Neustart
3Docker force-cleanup + Neustart

6 Zwischen Apps wechseln

Da jede App ihre eigenen Volumes und Ports hat, können mehrere Apps gleichzeitig laufen. Beim Wechsel einfach die andere App starten – laufende Apps werden nicht automatisch gestoppt.

# App A stoppen
.\scripts\stop.ps1 -App auktivo

# App B starten
.\scripts\setup.ps1 -App parzella
Paralleler Betrieb möglich

Da jedem App-Namen ein eigener Port-Block zugewiesen wird, können Apps A und B gleichzeitig laufen. Die Port-Zuweisungen werden in .ports persistent gespeichert.

7 App-Daten zurücksetzen

Der -Reset-Parameter löscht das Docker-Volume der App vollständig und startet mit einer frischen Datenbank – inklusive Migrations und Seed.

# Alle Daten löschen und neu starten
.\scripts\setup.ps1 -App auktivo -Reset
Unwiderruflich

Ein Reset löscht das Docker-Volume supabase_db_<app> dauerhaft. Alle manuell eingefügten Daten gehen verloren. Migrations und Seed-Daten werden automatisch neu eingespielt.

Wann ist ein Reset notwendig?

  • Nach einem Upgrade der PostgreSQL-Hauptversion (z. B. 15 → 17)
  • Bei einem korrupten Datenbankzustand
  • Um mit sauberem Datenbankzustand neu zu beginnen
  • Nach Änderungen an Migrationen, die einen sauberen Durchlauf erfordern

8 Alles löschen (Purge)

Der Purge-Befehl entfernt alle Supabase-Ressourcen aller Apps auf diesem Rechner – Volumes, Container und Netzwerke.

# Windows
.\scripts\purge.ps1

# macOS / Linux
./scripts/purge.sh
Achtung – alle Apps betroffen

Purge löscht die Daten aller Supabase-Apps auf dem lokalen Rechner, nicht nur der aktuell aktiven. Nach einem Purge müssen alle Apps mit setup.ps1 -App <name> neu initialisiert werden.

9 Migrations

Datenbankmigrationen liegen in supabase/migrations/ als .sql-Dateien mit Timestamp-Präfix. Die Supabase CLI spielt sie beim Start automatisch in der richtigen Reihenfolge ein.

Neue Migration erstellen

# Supabase CLI erstellt eine neue leere Migration
supabase migration new create_users_table

# Ergebnis: supabase/migrations/20260516120000_create_users_table.sql

Migration manuell anwenden

# Alle ausstehenden Migrations einspielen
supabase db push

# Status anzeigen
supabase migration list

Seed-Daten

Initiale Testdaten werden aus supabase/seed.sql automatisch beim ersten Start (oder nach einem Reset) eingespielt.

Migrations werden beim Setup automatisch angewendet

setup.ps1 ruft intern supabase start auf, welches automatisch alle Migrations und den Seed einspielt – kein manueller Schritt erforderlich.

11 Wechsel zu Supabase Cloud

Wenn die App produktionsreif ist, kann auf Supabase Cloud umgestellt werden. Die Migrations aus supabase/migrations/ können direkt auf ein Cloud-Projekt angewendet werden.

  1. Supabase Cloud Projekt erstellen unter app.supabase.com
  2. Projekt verknüpfen 
    supabase link --project-ref dein-projekt-ref
  3. Migrations hochladen 
    supabase db push
  4. .env.local mit den Cloud-URLs und -Keys aus dem Supabase Dashboard aktualisieren
Migrations sind Cloud-kompatibel

Da alle Migrations als Standard-SQL in supabase/migrations/ liegen, können sie ohne Änderungen auf Cloud-Projekte angewendet werden.

13 Troubleshooting

Docker läuft nicht

Fehlermeldung

Docker läuft nicht. Bitte Docker Desktop starten.

Lösung: Docker Desktop starten und warten, bis der Daemon vollständig hochgefahren ist (docker ps sollte ohne Fehler antworten).

Port bereits belegt

Symptom

Ein Dienst startet nicht, weil der Port bereits durch eine andere Anwendung belegt ist.

Lösung: Das Setup-Skript sucht automatisch nach freien Port-Blöcken. Falls trotzdem ein Konflikt auftritt, den Prozess auf dem belegten Port beenden:

# Port 54321 freigeben (Windows)
netstat -ano | findstr :54321
taskkill /PID <PID> /F

PostgreSQL-Versions-Konflikt

Fehlermeldung

FATALER FEHLER: PostgreSQL-Versions-Konflikt!
Das Docker-Volume enthält eine andere PostgreSQL-Version als in config.toml konfiguriert (z. B. PG 15 vs. PG 17).

Option 1: Daten sichern, dann Reset

# 1. Daten exportieren
docker run --rm -v supabase_db_auktivo:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=postgres -p 5434:5432 -d \
  --name pgexport postgres:15
docker exec pgexport pg_dumpall -U postgres > backup.sql
docker stop pgexport

# 2. Reset mit neuer PG-Version
.\scripts\setup.ps1 -App auktivo -Reset

Option 2: Reset ohne Backup (Daten werden gelöscht)

.\scripts\setup.ps1 -App auktivo -Reset

Supabase CLI gibt non-zero Exit Code zurück, Dienste laufen aber

Manche CLI-Versionen geben beim Start einen Fehlercode zurück, obwohl alle Dienste korrekt laufen. Das Setup-Skript erkennt diesen Fall automatisch und ignoriert den Exit Code, wenn die Dienste tatsächlich erreichbar sind.

Keys können nicht ermittelt werden

Nach dem Start versucht das Skript bis zu 3 Mal (mit 5 Sekunden Pause), die Keys abzurufen. Schlägt dies fehl:

# Status manuell prüfen
supabase status
supabase status -o env
